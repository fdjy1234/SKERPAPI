## 2. Fix-2 技術說明

### 問題根因

`RbacAuthorizeAttribute.OnActionExecutingAsync` 原本只讀取 `X-Api-Key` HTTP Header 作為身份識別來源，完全忽略 OWIN Layer 1 中介層（`ApiKeyAuthMiddleware`、`OAuthBearerMiddleware`、`ClientCertificateAuthMiddleware`）設定的 `ClaimsPrincipal`。

**影響**：使用 OAuth 2.0 Bearer Token 或 mTLS 憑證的用戶端，在 OWIN Layer 1 成功認證後，仍被 Layer 2 RBAC 以 401 拒絕，因為它們沒有 `X-Api-Key` Header。

### 修正方案

修改 `src/SKERPAPI.Core/Filters/RbacAuthorizeAttribute.cs`，實作 `clientId` 三層解析邏輯：

| 優先順序 | 來源 | 適用場景 |
|---|---|---|
| 1 | `ClaimsPrincipal.FindFirst("client_id")` | OAuth Bearer Token、mTLS、API Key（皆由 Layer 1 設定） |
| 2 | `ClaimsPrincipal.FindFirst(ClaimTypes.Name)` | 備用 Name Claim |
| 3 | `X-Api-Key` Header | 向後相容舊版 API Key 用戶端（無 OWIN 認證） |
| 4 | 三者皆空 | → **401 Unauthorized** |
