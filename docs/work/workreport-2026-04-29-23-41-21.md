## 一、系統認證架構總覽

### OWIN 中介層管線（由外到內）

```
HTTP(S) Request
    │
    ▼  Layer 0 ── CORS Middleware
    │               (preflight OPTIONS 處理)
    │
    ▼  Layer 1a ── ClientCertificateAuthMiddleware   ← mTLS
    │               src/SKERPAPI.Core/Security/Authentication/ClientCertificateAuthMiddleware.cs
    │               被動式 (Passive)：驗證失敗不立即拒絕，除非 MtlsRequired=true
    │
    ▼  Layer 1b ── OAuthAuthorizationServer          ← OAuth 2.0 Token 端點
    │               POST /api/token
    │               OAuthBearerAuthentication         ← Bearer Token 驗證
    │
    ▼  Layer 1c ── ApiKeyAuthMiddleware               ← Static API Key
    │               src/SKERPAPI.Core/Security/Authentication/ApiKeyAuthMiddleware.cs
    │               被動式 (Passive)：驗證失敗不立即拒絕
    │
    ▼  Layer 1d ── AuthenticationRequiredMiddleware   ← 認證閘門
    │               確保 1a/1b/1c 至少一個通過；否則 401
    │               白名單：/api/token、/swagger、/api/health
    │
    ▼  Web API Pipeline (Routing + Filters)
    │
    ▼  Layer 2  ── RbacAuthorizeAttribute             ← 第二層授權
                    [RbacAuthorize(Permission = "aoi:xxx:yyy")]
                    呼叫 IRbacService → CachingRbacService → RbacServiceClient
                    GET http://internal-rbac-api:8080/api/rbac/keys/{apiKey}
```

## 二、Static API Key 評估

### 2.1 實作位置

| 元件 | 路徑 | 說明 |
|------|------|------|
| `ApiKeyAuthMiddleware` | `Security/Authentication/ApiKeyAuthMiddleware.cs` | OWIN 主要實作 |
| `ApiKeyAttribute` | `Core/Filters/ApiKeyAttribute.cs` | 已棄用 (Obsolete)，保留相容 |
| Web.config | `Security:ApiKeys` | 多金鑰設定（clientId:key 格式）|

### 2.2 支援的模式

```
模式 1 (多金鑰)：Security:ApiKeys = "client-aoi:key123,client-car:key456"
  → 格式解析：Split(',') → Split(':', 2)
  → ClientId = "client-aoi"

模式 2 (單一金鑰 / 向後相容)：ApiKey = "single-key"
  → ClientId = "default-client"

模式 3 (開發環境)：兩個 Key 皆為空
  → IsValid = true, ClientId = "dev-client"  ← ⚠️ 安全風險
```

### 2.3 認證流程

```
X-Api-Key: client-aoi:key123
    │
    ▼ ApiKeyAuthMiddleware.Invoke()
    │   ValidateApiKey(apiKey)
    │   → ClaimsIdentity(ApiKeyAuthentication)
    │   → ClaimTypes.Name = "client-aoi"
    │   → ClaimTypes.AuthenticationMethod = "ApiKey"
    ▼
  ClaimsPrincipal 設定完成
    │
    ▼ AuthenticationRequiredMiddleware → 通過
    │
    ▼ RbacAuthorizeAttribute
        再次讀取 X-Api-Key Header
        → RbacServiceClient.ResolveAsync(apiKey)
        → GET /api/rbac/keys/{apiKey} → {Permissions:[...]}
        → HasPermission("aoi:inspection:execute") ? 200 : 403
```

### 2.4 問題清單

| # | 嚴重性 | 問題 | 位置 |
|---|--------|------|------|
| A-1 | 🔴 HIGH | **開發模式全放行**：`Security:ApiKeys` 和 `ApiKey` 皆為空時，任何不帶金鑰的請求都被認證為 `dev-client`，若此設定誤部署到生產環境將完全開放 | `ApiKeyAuthMiddleware.cs` L89-93 |
| A-2 | 🔴 HIGH | **Layer 2 重複讀取 Header**：`RbacAuthorizeAttribute` 不使用 Layer 1 設定的 `ClaimsPrincipal`，而是再次讀取原始 `X-Api-Key` Header，導致雙重驗證且架構耦合 | `RbacAuthorizeAttribute.cs` L51-57 |
| A-3 | 🟠 MEDIUM | **金鑰明文儲存**：API Keys 儲存在 `Web.config` appSettings 純文字，未加密。若 Web.config 外洩則所有金鑰曝光 | `Web.config` |
| A-4 | 🟠 MEDIUM | **無速率限制**：暴力破解 API Key 無防護機制（無 IP 封鎖、無嘗試計數） | OWIN Pipeline |
| A-5 | 🟡 LOW | **非常數時間比較**：`parts[1].Trim() == apiKey` 為字串相等比較，理論上存在 Timing Attack 風險（在本地驗證中影響較小） | `ApiKeyAuthMiddleware.cs` L79 |
| A-6 | 🟡 LOW | **舊 Attribute 殘留**：`ApiKeyAttribute` 雖標記 Obsolete，若有舊 Controller 仍使用則繞過 OWIN 管線的統一管理 | `ApiKeyAttribute.cs` |

### 2.5 整合完整性

```
Layer 1 → Layer 2 整合狀態：⚠️ 部分整合
  ✅ Layer 1 設定 ClaimsPrincipal（可被 PermissionAuthorizeAttribute 使用）
  ❌ Layer 2 (RbacAuthorizeAttribute) 忽略 ClaimsPrincipal，重新讀取 Header
  ❌ 無 Key Rotation 機制：更換 Key 需重啟服務（Web.config 變更）
```

## 三、OAuth 2.0 評估

### 3.1 實作位置

| 元件 | 路徑 | 說明 |
|------|------|------|
| `OAuthServerConfig` | `Security/Authentication/OAuthServerConfig.cs` | Token 端點配置 |
| `SkerpApiOAuthProvider` | 同上 | Grant Type 實作 |
| `OAuthBearerAuthentication` | `Startup.cs` | Bearer Token 驗證 |

### 3.2 支援的 Grant Types

```
POST /api/token
Content-Type: application/x-www-form-urlencoded

─ client_credentials ────────────────────────────────── ✅ 實作
  grant_type=client_credentials
  &client_id=client-aoi
  &client_secret=key123          ← 複用 API Key 驗證邏輯
  → 200 { access_token, token_type: "bearer", expires_in: 3600 }

─ password (ROPC) ───────────────────────────────────── ⚠️ 部分實作
  grant_type=password
  &username=user01
  &password=pass01               ← TODO: 無實際驗證，任何帳密皆通過
  → 200 { access_token, ... }

─ authorization_code ────────────────────────────────── ❌ 未實作（API-to-API 場景不需要）
─ refresh_token ─────────────────────────────────────── ❌ 未實作
```

### 3.3 Token 格式與安全性

```
AccessTokenFormat = null  →  使用 Microsoft.Owin Machine Key 保護的 Opaque Token

優點：
  ✅ 不需要 JWT Secret 管理
  ✅ Token 自動與 Machine Key 綁定

缺點：
  ❌ 非 JWT 格式：無法被外部服務驗證（無 introspection endpoint）
  ❌ 無法水平擴展：多台伺服器 Machine Key 必須同步
  ❌ 無 Claims 可攜性：Token Payload 不透明，無法攜帶 Permission Claims
  ❌ Web.config Jwt:Secret = "" (空)：JWT 模式未啟用
```

### 3.4 問題清單

| # | 嚴重性 | 問題 | 位置 |
|---|--------|------|------|
| O-1 | 🔴 HIGH | **Password Grant 無驗證**：`GrantResourceOwnerCredentials` 標記 TODO，任何 username/password 組合都會發行 Token | `OAuthServerConfig.cs` L145-161 |
| O-2 | 🔴 HIGH | **Layer 2 無法驗證 Bearer Token**：`RbacAuthorizeAttribute` 讀取 `X-Api-Key` Header，Bearer Token 持有者沒有 API Key Header 時，Layer 2 會回傳 401，即使 Layer 1 已認證成功 | `RbacAuthorizeAttribute.cs` L51-57 |
| O-3 | 🟠 MEDIUM | **Opaque Token 不可攜**：Token 只能在同一台伺服器驗證，無 Token Introspection 端點，無法支援微服務間驗證 | `OAuthServerConfig.cs` L56 |
| O-4 | 🟠 MEDIUM | **AllowInsecureHttp 依賴 DEBUG 編譯**：正式部署必須確認以 Release 模式編譯，否則 HTTP 通道可能被允許 | `OAuthServerConfig.cs` L47-50 |
| O-5 | 🟠 MEDIUM | **無 Refresh Token**：Token 過期（預設 60 分鐘）後需重新申請，無 Silent Refresh 機制，對長時間運行的自動化系統不友好 | - |
| O-6 | 🟡 LOW | **client_id 非強制驗證**：`ValidateClientAuthentication` 在 `clientId` 為空時直接 `Validated()`，允許匿名 password grant | `OAuthServerConfig.cs` L109-113 |

### 3.5 整合完整性

```
Layer 1 → Layer 2 整合狀態：❌ 未整合
  ✅ Bearer Token 由 OAuthBearerAuthentication 驗證，設定 ClaimsPrincipal
  ❌ RbacAuthorizeAttribute 僅讀取 X-Api-Key，Bearer Token 客戶端無法通過 Layer 2
  ❌ PermissionAuthorizeAttribute 可搭配使用（ClaimsPrincipal），但需手動同步權限
```

## 四、mTLS（Mutual TLS）評估

### 4.1 實作位置

| 元件 | 路徑 | 說明 |
|------|------|------|
| `ClientCertificateAuthMiddleware` | `Security/Authentication/ClientCertificateAuthMiddleware.cs` | OWIN 主要實作 |
| Web.config | `Security:MtlsRequired`, `Security:MtlsTrustedIssuers` | 設定 |

### 4.2 驗證邏輯流程

```
TLS Handshake (IIS 層)
    │  IIS SSL Settings: "Accept" or "Require"
    │  Client Certificate → ssl.ClientCertificate (OWIN context)
    ▼
ClientCertificateAuthMiddleware.Invoke()
    │
    ├─ 步驟 1：有效期驗證
    │   DateTime.Now < NotBefore || DateTime.Now > NotAfter → 拒絕
    │
    ├─ 步驟 2：Windows CA Chain 驗證
    │   X509Chain.Build(certificate)
    │   RevocationMode = Online  ← CRL/OCSP 線上查詢
    │   RevocationFlag = EntireChain
    │   VerificationFlags = NoFlag（嚴格模式）
    │
    ├─ 步驟 3：Issuer 白名單（可選）
    │   Security:MtlsTrustedIssuers = "CN=Company-CA,CN=Corp-SubCA"
    │   空值 = 信任所有 CA Store 中的 CA
    │
    └─ 成功 → ClaimsIdentity(ClientCertificateAuthentication)
               Claim: Name = certificate.Subject (完整 DN)
               Claim: cert_thumbprint = certificate.Thumbprint
               Claim: cert_issuer = certificate.Issuer
               Claim: AuthenticationMethod = "ClientCertificate"
```

### 4.3 問題清單

| # | 嚴重性 | 問題 | 位置 |
|---|--------|------|------|
| M-1 | 🔴 HIGH | **Layer 2 完全不支援 mTLS**：`RbacAuthorizeAttribute` 讀取 `X-Api-Key`，mTLS 客戶端沒有 API Key 時 Layer 2 回傳 401，mTLS 認證形同虛設（授權層被繞過） | `RbacAuthorizeAttribute.cs` L51-57 |
| M-2 | 🔴 HIGH | **MtlsRequired 預設 false**：`Security:MtlsRequired = false`，mTLS 完全為可選，若預期所有 API 呼叫都需要憑證則設定必須改為 true | `Web.config` |
| M-3 | 🟠 MEDIUM | **IIS 強依賴**：需要在 IIS → SSL Settings 設定接受 Client Certificate，此設定在 IIS Express/Kestrel 環境無效，整合測試困難 | IIS 設定 |
| M-4 | 🟠 MEDIUM | **Online CRL 在隔離環境失效**：`X509RevocationMode.Online` 在無網路的工廠環境或 Air-gapped DMZ 會使 Chain Validation 失敗 | `ClientCertificateAuthMiddleware.cs` L122 |
| M-5 | 🟠 MEDIUM | **Subject DN 作為 UserId**：憑證 Subject = `"CN=Factory-Robot01,O=Corp,C=TW"` 作為 ClaimTypes.Name，此格式與 RBAC 的 UserId/ClientId 格式不一致，無法直接對應 RolePermission | `ClientCertificateAuthMiddleware.cs` L51 |
| M-6 | 🟡 LOW | **無憑證固定 (Certificate Pinning)**：僅驗證 CA 鏈，未限制特定憑證 Thumbprint，若企業 CA 被入侵則任何該 CA 簽發的憑證均可通過 | `ClientCertificateAuthMiddleware.cs` |
| M-7 | 🟡 LOW | **Revocation 非同步阻塞**：`X509Chain.Build()` 為同步呼叫且需網路 I/O，可能在高並發下造成執行緒阻塞 | `ClientCertificateAuthMiddleware.cs` L118 |

### 4.4 整合完整性

```
Layer 1 → Layer 2 整合狀態：❌ 未整合
  ✅ ClaimsPrincipal 設定完成（含 cert_thumbprint, cert_issuer）
  ❌ RbacAuthorizeAttribute 完全不讀取 ClaimsPrincipal，mTLS 客戶端無法通過 Layer 2
  ❌ 無 Certificate CN → RBAC ClientId 的 Mapping 機制
  ❌ PermissionAuthorizeAttribute 可搭配（ClaimsPrincipal 已設定），但需手動配置
```

## 七、根本問題（架構層面）

### 問題核心：Layer 2 (`RbacAuthorizeAttribute`) 硬耦合 API Key

```csharp
// 現況：僅支援 API Key
if (actionContext.Request.Headers.TryGetValues("X-Api-Key", out ...))
    apiKey = headerValues.FirstOrDefault();

// 缺少的設計：
// 1. 從 ClaimsPrincipal 取得 ClientId（支援所有認證方式）
// 2. ClientId → Permission 的解析不應只依賴 X-Api-Key Header
```

**影響：**
- OAuth 2.0 Bearer Token 客戶端 → Layer 2 回傳 **401**（即使 Layer 1 通過）
- mTLS 客戶端 → Layer 2 回傳 **401**（即使憑證合法）
- 實質上只有 API Key 是一個「完整」的認證路徑

## 八、修復路線圖（優先序）

### Phase 1（立即，1-2 天）—— 修正高危問題

#### Fix-1：關閉生產環境開發模式 (A-1)

```csharp
// ApiKeyAuthMiddleware.cs — 移除開發模式全放行邏輯
// 改為：未設定金鑰時只允許已認證請求通過（由後續中介層處理）
if (string.IsNullOrEmpty(singleKey))
{
    // 不設定 ClaimsPrincipal，讓 AuthenticationRequiredMiddleware 決定
    return new ApiKeyValidationResult { IsValid = false };
}
```

#### Fix-2：Layer 2 改為讀取 ClaimsPrincipal (A-2, O-2, M-1)

```csharp
// RbacAuthorizeAttribute.cs — 統一認證身份來源
public override async Task OnActionExecutingAsync(...)
{
    // 優先使用 Layer 1 已建立的 ClaimsPrincipal
    var principal = actionContext.RequestContext.Principal as ClaimsPrincipal;
    var clientId = principal?.FindFirst(SecurityConstants.ClientIdClaimType)?.Value
                ?? principal?.FindFirst(ClaimTypes.Name)?.Value;

    // Fallback：向後相容，嘗試讀取 X-Api-Key（僅當 ClaimsPrincipal 不存在時）
    if (string.IsNullOrEmpty(clientId))
    {
        if (actionContext.Request.Headers.TryGetValues("X-Api-Key", out var vals))
            clientId = vals.FirstOrDefault();
    }

    if (string.IsNullOrEmpty(clientId))
    {
        SetResponse(actionContext, HttpStatusCode.Unauthorized, "Missing authentication.");
        return;
    }
    // ... 後續用 clientId 查詢 RBAC
}
```

#### Fix-3：Password Grant 加入驗證 (O-1)

```csharp
// OAuthServerConfig.cs — GrantResourceOwnerCredentials
// 必須對接企業 LDAP/RBAC 驗證 username/password
// 過渡期：加入拒絕邏輯，不允許此 grant type
context.SetError("unsupported_grant_type", "Password grant is not enabled.");
return Task.CompletedTask;
```

### Phase 2（本週，3-5 天）—— 架構完善

#### Fix-4：API Key 加密儲存

使用 `<appSettings configProtectionProvider="DataProtectionConfigurationProvider">` 或 Azure Key Vault 加密 Web.config appSettings。

#### Fix-5：mTLS Certificate CN → ClientId Mapping

```csharp
// 在 ClientCertificateAuthMiddleware 中新增：
// 從 Subject DN 提取 CN，作為標準化的 ClientId
var cn = ExtractCN(certificate.Subject);  // "CN=Factory-Robot01,..." → "Factory-Robot01"
identity.AddClaim(new Claim(SecurityConstants.ClientIdClaimType, cn));
```

#### Fix-6：速率限制

在 `AuthenticationRequiredMiddleware` 或獨立 Middleware 中加入 IP + ClientId 基礎的速率限制（可用 `System.Runtime.Caching.MemoryCache` 實作滑動窗口計數器）。

### Phase 3（下週）—— OAuth 生產化

#### Fix-7：切換 JWT Token 格式

```xml
<!-- Web.config -->
<add key="Jwt:Secret" value="[至少 32 位元組的隨機字串]" />
```

```csharp
// OAuthServerConfig.cs — 使用 JWT 格式 Token
AccessTokenFormat = new JwtFormat(
    audience: ConfigurationManager.AppSettings["Jwt:Audience"],
    issuerCredentialProvider: new SymmetricKeyIssuerSecurityKeyProvider(
        issuer: ConfigurationManager.AppSettings["Jwt:Issuer"],
        key: Encoding.UTF8.GetBytes(ConfigurationManager.AppSettings["Jwt:Secret"])
    )
);
```

#### Fix-8：mTLS Revocation 策略調整（工廠環境）

```csharp
// 支援可設定的撤銷模式
var revocationMode = ConfigurationManager.AppSettings["Security:MtlsRevocationMode"];
chain.ChainPolicy.RevocationMode = revocationMode == "Offline"
    ? X509RevocationMode.Offline
    : X509RevocationMode.Online;
```

## 九、各認證方法適用場景建議

| 場景 | 建議認證方式 | 備注 |
|------|-------------|------|
| 服務間自動化呼叫（MES, ERP） | Static API Key | 修正 A-1, A-3 後可用；考慮移至 Secrets Manager |
| 工廠設備 / IoT 機台 | mTLS | 修正 M-1, M-3, M-5 後最安全；憑證由企業 CA 統一發行 |
| 第三方合作夥伴系統 | OAuth 2.0 client_credentials | 修正 O-1, O-3 後可用；建議改用外部 IdP（Keycloak/Azure AD）|
| 人工操作介面（Console/Postman） | OAuth 2.0 password grant | 修正 O-1 後可用；長期建議換成 authorization_code + PKCE |
| 內部開發測試 | Static API Key（本地模式）| 確保 dev 模式不進入 staging/production 環境 |

## 十、結論

SKERPAPI 的三種認證方式在 **Layer 1 (OWIN Pipeline)** 均已有完整框架，設計思路清晰且符合 OWIN 被動認證 (Passive Authentication) 最佳實踐。然而，**Layer 2 (`RbacAuthorizeAttribute`) 僅支援 API Key**，導致 OAuth 2.0 和 mTLS 認證的客戶端在授權層失效，是目前最嚴重的架構問題。

**最優先修正：Fix-2（Layer 2 改為讀取 ClaimsPrincipal）**，此單一修改可同時解決三種認證方式的 Layer 2 整合問題，工程量小、影響面廣。

```
整體評估：
  Static API Key  ——  🟡 可用，需修正 A-1（開發模式洩漏）+ A-2（重複驗證）
  OAuth 2.0       ——  🟠 框架完整，需修正 O-1（無驗證 Password Grant）+ O-2（Layer 2 整合）
  mTLS            ——  🟠 框架完整，需修正 M-1（Layer 2 整合）+ M-5（Subject DN 正規化）
  跨層整合        ——  🔴 缺失，Layer 2 僅支援 API Key，Fix-2 為最高優先修正項
```
